Front end workflows, atomic design, reusable components

Enter Pattern Lab

At its core, Pattern Lab is a static site generator (powered by either PHP or Node) that stitches together UI components. But there's a whole lot more to it than that!

Pattern Lab as a design system

  • Build pattern driven user interfaces using atomic design principles
  • Include patterns inside each other to create compenets from consistent building blocks
  • Pattern documentation - have the entire team using the same language throughout the entire project

Pattern Lab as an implementation

  • PHP or node version
  • Tool agnostic
  • Language agnostic
  • Templating engine flexible (Mustache, Twig, Handlebars, etc)
  • Viewport resizer, annotations, pattern lineage, and more

How it works

Create templates, starting with the smallest pieces. Use those patterns to create larger pieces. Again and again.

You do not need to include all of your data statically. That would be not ideal, because you might want to use a specific template many times within your Pattern Lab project with different data.

Each template file is also backed by a JSON file that provides the content for the template.

In short, any time you include a template, you can customize the data being displayed if you desire.

Pattern Lab in Drupal

Using Pattern Lab in a Drupal 7 project would require maintaining a separate Pattern Lab project. Using same CSS/JS files would require manually maintaining the consistency of templates between the two projects.

Because Drupal 8 uses Twig as its templating engine, the opportunity to add Pattern Lab to your Drupal project is now a possibility.

Drupal 8 and Twig

  • Big change for frontend workflow in Drupal 8
  • Templating language that adds common templating patterns to PHP projects, learn more about Twig
  • All templates in Drupal 8 are .twig files
Challenges adding Pattern Lab to a Drupal 8 project

How it is different in this context than on its own

Pattern Lab implements a vanilla project based on whatever templating engine you choose to use. It assumes no dependencies. So, the general way to build more and more complex patterns is to include the simpler patterns within the more complex pattern templates.

For example


{% include "atoms-title" %}
{% include "atoms-landscape-16x9" %}
{% include "atoms-body" %}

If you're thinking: "Wait, but that's a problem, because my Drupal templates don't do that!" You're right, it's a problem! And it's in the way of reaching our gold standard goal.

WELL, now you might be thinking, WHAT IF I can somehow tell Pattern Lab about the patterns I want to include in this JSON file instead of in the template itself. That'd be swell!

Special JSON Keys

  • attributes() - Compiles the Drupal attributes
  • include() - Include a template in the specified variable
  • join() - Join multiple templates together in the specified variable

The JSON files are processed and these three special keys allow for specific behavior.

How/Why is this useful?
Decouples the front end from Drupal! You can theme almost completely OUTSIDE of Drupal if you want.
Potentially allows a front end to be functionally built out and designed before a committment to sitebuild is made without losing that time
Isolates components into the pieces of responsibility outside of the context of Drupal (wrappers, containers, other element classes, etc)
Produces a comprehensive styleguide asset!
Provides an entire set of data for all your front end components that does not depend on the database. Comprehensive demonstration of all styles for easy visual "regression testing" of sorts. Especially if you are building complicated components, or items that have many moving parts.
Connecting communities! Pattern Lab is an existing tool with a lot of dedicated followers. And with the hard work of a few people, a bridge to connect Drupal with this large existing community has been made. That's pretty cool!
Why may this be not ideal for every project?
Time - Though, through this talk, I hope we determined that with a reasonable setup, the time investment to use this workflow is not that great. For medium/large projects, or projects that are maintained, it can be well worth it, and save more than the time it takes to create
More Tools! - Yes, there are more tools to learn and more things to potentially debug. To effectively use this workflow, the dev has to have a reasonable knowledge of PHP, understanding of twig, a strong understanding of Drupal 8 conventions, experience with front end workflow tools (gulp/grunt), maybe some amount of experience with node, and able to debug any of those parts.
Too many tools? It depends on your workflow, and your resources. But this is the direction of the Front End developer in the web space. Tools beyond html/css/javascript are par for the course.
End to End project workflow - How does your company work? There are a LOT of potential benefits from this type of workflow, but if your entire project process doesn't/can't adjust to position itself to gain from these benefits than you may strictly be weighing the internal dev benefits against the dev costs.
